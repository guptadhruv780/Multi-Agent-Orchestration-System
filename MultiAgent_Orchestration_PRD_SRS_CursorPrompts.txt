1.1 EXECUTIVE SUMMARY
------------------------------------------------------------------------

The Full-Stack Multi-Agent Orchestration System is an autonomous AI-powered
software engineering platform that closes the gap between issue discovery and
code delivery. It ingests a raw GitHub Issue URL, decomposes the problem into
specialized sub-tasks, and — through a coordinated graph of AI agents —
researches the repository, drafts a contextually-aware fix, validates it
inside an isolated Docker sandbox, and opens a Pull Request, all without
human intervention.

The platform surfaces every agent action in real time via a React dashboard,
giving engineers complete observability over the autonomous workflow. Unlike
linear LLM chains, the system employs a stateful, cyclic LangGraph graph
that self-corrects: if generated tests fail, the graph loops back to the
Coding Agent with the error context, mimicking how a skilled engineer
iterates until tests pass.

Core Value Proposition:
  - Reduces time-to-PR for well-scoped bugs from hours to minutes.
  - Provides full auditability — every agent decision is logged and streamed.
  - Architected for extensibility — new agents (security review, doc writer)
    can be added as new LangGraph nodes without refactoring the core graph.

PRIMARY USERS:
  A. Individual Open-Source Maintainers
     - Scenario: A maintainer has 50+ open issues. They paste the issue URL
       into the dashboard and let the system propose a fix as a draft PR

     - Scenario: Internal tooling automation. The system is embedded in

     - Scenario: Benchmarking LLM coding capabilities on real-world repos
       using SWE-bench style evaluation.

  UC-01: Automated Bug Fix       - Issue describes a runtime error or wrong
                                   output; system patches the relevant file.
  UC-02: Test Gap Coverage       - Issue requests additional test coverage;
                                   system generates and validates tests.
  UC-03: Boilerplate Generation  - Issue requests a new CRUD endpoint or

  UC-04: Dependency Upgrade      - Issue flags a deprecated package; system
                                   updates imports and fixes breaking changes.
  UC-05: Documentation Fix       - Issue notes outdated docstrings or README

1.3 CORE FEATURES
------------------------------------------------------------------------

F-01: AGENT ROLES
------------------
  ORCHESTRATOR AGENT
    - Role: Central controller / router. Does NOT perform coding tasks.
    - Responsibilities: Reads current graph state, evaluates conditional
      edges, decides next node to activate (Research → Code → Test → PR
      or Code → Test → Code loop).
    - Tool Access: State inspection only.

  RESEARCH AGENT
    - Role: Autonomous codebase analyst.
    - Responsibilities: Fetches issue metadata, clones/browses repo via
      GitHub API, identifies relevant files, methods, and dependencies,
      builds a structured "context brief" for the Coding Agent.
    - Tool Access: search_codebase(), read_file(), list_directory(),
      get_issue_details(), get_repo_structure().

  CODING AGENT
    - Role: Code generation and patch creation.
    - Responsibilities: Receives context brief + (on retry) test failure
      logs. Produces a unified diff / concrete file patches. On iteration N>1,
      analyzes the test error and refines its approach.
    - Tool Access: read_file(), apply_patch(), write_file().

  TESTING AGENT
    - Role: Sandboxed validation gatekeeper.
    - Responsibilities: Takes the patched files, spins up an ephemeral
      Docker container with the project's environment, runs the existing
      test suite + any newly generated tests, returns structured pass/fail
      results with full stdout/stderr.
    - Tool Access: run_docker_container(), read_test_output(),
      install_dependencies().

  PR AGENT
    - Role: GitHub integration finisher.
    - Responsibilities: Creates a feature branch, commits patched files,
      opens a Pull Request with a structured description (problem, solution,
      test evidence), and returns the PR URL.
    - Tool Access: create_branch(), commit_files(), open_pull_request(),
      add_pr_comment().

F-02: STATEFUL CYCLIC WORKFLOW (SELF-CORRECTION LOOP)
------------------------------------------------------
  The graph operates on a persistent State object (TypedDict) that is
  mutated and passed between nodes. The critical loop is:

    [START]
       │
       ▼
  [Research Agent] ──────────────────────────────────────────────────────┐
       │                                                                  │
       ▼                                                                  │
  [Coding Agent] ◄─────────────────────────────────────┐                 │
       │                                                │                 │
       ▼                                                │                 │
  [Testing Agent]                                       │                 │
       │                                                │                 │
       ├──── TESTS PASS ──────────────────────────────────────────────────┤
       │                                                │                 │
       └──── TESTS FAIL (retry_count < MAX_RETRIES) ───┘                 │
       │                                                                  │
       └──── TESTS FAIL (retry_count >= MAX_RETRIES) ──► [FAIL STATE]    │
                                                                          │
  [PR Agent] ◄──────────────────────────────────────────────────────────┘
       │
       ▼
    [END]

  State propagation rule: Each node receives the FULL state, appends its
  output to the relevant fields, and returns the updated state. LangGraph
  handles merging via its add_messages reducer pattern.

LANGGRAPH NODE/EDGE DETAILED STRUCTURE:
========================================

  Graph Definition (Python pseudocode):

    graph = StateGraph(AgentState)
    graph.add_node("research",  research_node_fn)
    graph.add_node("coding",    coding_node_fn)
    graph.add_node("testing",   testing_node_fn)
    graph.add_node("pr",        pr_node_fn)

    graph.set_entry_point("research")
    graph.add_edge("research", "coding")
    graph.add_edge("coding",   "testing")
    graph.add_conditional_edges(
        "testing",
        route_after_testing,          # conditional router function
        {
            "pass":  "pr",
            "retry": "coding",
            "fail":  END
        }
    )
    graph.add_edge("pr", END)

    checkpointer = MemorySaver()      # or SqliteSaver for persistence
    app = graph.compile(checkpointer=checkpointer)

  CONDITIONAL ROUTER LOGIC:

    def route_after_testing(state: AgentState) -> str:
        if state["test_result"]["status"] == "pass":
            return "pass"
        if state["retry_count"] < MAX_RETRIES:
            return "retry"
        return "fail"

2.3 STATE SCHEMA DEFINITION
------------------------------------------------------------------------

  # File: backend/core/state.py

  from typing import TypedDict, Annotated, Optional, List
  from operator import add
  import operator

  class FileChange(TypedDict):
      file_path: str            # Relative path in repo
      original_content: str     # Content before patch
      patched_content: str      # Content after patch
      diff: str                 # Unified diff string

  class TestResult(TypedDict):
      status: str               # "pass" | "fail" | "error"
      total_tests: int
      passed: int
      failed: int
      errors: int
      stdout: str               # Full test runner output
      stderr: str               # Error output from container
      exit_code: int            # Docker container exit code

  class AgentLog(TypedDict):
      agent_name: str           # "research" | "coding" | "testing" | "pr"
      level: str                # "info" | "warning" | "error"
      message: str
      timestamp: str            # ISO 8601

  class AgentState(TypedDict):
      # --- INPUT FIELDS (set once at graph entry) ---
      run_id:          str                      # UUID for this execution
      issue_url:       str                      # Full GitHub issue URL
      repo_owner:      str                      # Parsed from issue_url
      repo_name:       str                      # Parsed from issue_url
      issue_number:    int                      # Parsed from issue_url
      github_token:    str                      # PAT for API calls
      model_name:      str                      # e.g. "claude-sonnet-4"

      # --- RESEARCH AGENT OUTPUT ---
      issue_title:     Optional[str]
      issue_body:      Optional[str]
      relevant_files:  Optional[List[str]]      # List of file paths
      codebase_context: Optional[str]           # Formatted context brief
      repo_structure:  Optional[str]            # Directory tree string

      # --- CODING AGENT OUTPUT ---
      file_changes:    Optional[List[FileChange]]
      patch_summary:   Optional[str]            # Human-readable change desc

      # --- TESTING AGENT OUTPUT ---
      test_result:     Optional[TestResult]
      retry_count:     int                      # Default 0, incremented on loop

      # --- PR AGENT OUTPUT ---
      branch_name:     Optional[str]
      pr_url:          Optional[str]
      pr_number:       Optional[int]

      # --- SHARED / SYSTEM FIELDS ---
      # Annotated[List, add] enables LangGraph to APPEND logs (not overwrite)
      logs:            Annotated[List[AgentLog], add]
      current_node:    str                      # Updated by each node
      final_status:    str                      # "running"|"success"|"failed"
      error_message:   Optional[str]            # Set on unhandled exceptions

  STATE MUTATION RULES:
    - Nodes must return ONLY the fields they modify (partial dict update).
    - The `logs` field uses the `add` reducer — all agents append to it.
    - The `retry_count` field is incremented in the router, not in nodes.
    - `current_node` is set at the START of each node function for live
      streaming awareness.

2.4 AGENT PROMPTS & TOOLING
------------------------------------------------------------------------

  RESEARCH AGENT SYSTEM PROMPT:
  --------------------------------
  You are an expert software engineering analyst. Your task is to fully
  understand a GitHub issue and the relevant portion of the codebase so
  that a Coding Agent can fix the issue without additional context.

  You will be given:
    - The full text of a GitHub issue (title + body)
    - The top-level repository structure

  Your job:
    1. Identify the ROOT CAUSE of the issue based on the description.
    2. Identify WHICH FILES are most likely responsible for the bug.
    3. Read those files using the available tools.
    4. Identify the EXACT functions, classes, or lines that need to change.
    5. Check for RELATED TESTS that already exist for this code path.
    6. Produce a structured "Context Brief" in this exact JSON format:
       {
         "root_cause_analysis": "...",
         "files_to_modify": ["path/to/file.py", ...],
         "relevant_code_snippets": {"path/to/file.py": "...snippet..."},
         "existing_tests": ["tests/test_file.py"],
         "suggested_approach": "...",
         "dependencies_to_check": ["module_name", ...]
       }

  RESEARCH AGENT TOOLS:
    - get_issue_details(owner, repo, issue_number) → dict
    - get_repo_structure(owner, repo, path="") → str (directory tree)
    - read_file(owner, repo, file_path, ref="main") → str (file content)
    - search_codebase(owner, repo, query) → List[dict] (code search results)
    - list_directory(owner, repo, path) → List[str]

  ---

  CODING AGENT SYSTEM PROMPT:
  --------------------------------
  You are an expert Python software engineer. Your task is to write the
  minimal, correct code fix for a GitHub issue based on a provided
  Context Brief.

  You will be given:
    - A structured Context Brief from the Research Agent (JSON)
    - The full content of relevant files
    - [On retry] The test failure output from the previous attempt

  Rules:
    1. Make the MINIMAL change necessary to fix the issue. Do NOT refactor
       unrelated code.
    2. Follow the existing code style (PEP 8, naming conventions,
       docstring format).
    3. If the issue requests new functionality, write tests for it.
    4. ALWAYS return your output in this EXACT JSON format:
       {
         "patch_summary": "One-sentence description of the fix",
         "file_changes": [
           {
             "file_path": "relative/path/to/file.py",
             "patched_content": "...FULL file content after fix...",
             "change_description": "What was changed and why"
           }
         ],
         "new_test_file": {
           "file_path": "tests/test_new_feature.py",
           "content": "...test file content..."
         }
       }

  [ON RETRY - append to prompt]:
    IMPORTANT: Your previous fix failed the test suite. Here is the
    failure output:
    --- TEST FAILURE LOG ---
    {test_result.stdout}
    {test_result.stderr}
    --- END LOG ---
    Analyze the failure, identify what was wrong in your previous patch,
    and produce a corrected fix.

  CODING AGENT TOOLS:
    - read_file(owner, repo, file_path, ref="main") → str
    - apply_patch(file_path, patched_content) → bool (writes to temp dir)
    - write_file(file_path, content) → bool (writes to temp workspace)

  ---

  TESTING AGENT SYSTEM PROMPT:
    (Testing Agent is primarily tool-driven, not LLM-driven in v1.0.
     It uses deterministic logic to run Docker and parse output.)

  TESTING AGENT TOOLS:
    - prepare_workspace(file_changes) → str (temp dir path)
    - build_docker_image(workspace_path, base_image) → str (image_id)
    - run_docker_container(image_id, test_command, timeout=120) → dict
      Returns: {exit_code, stdout, stderr}
    - parse_pytest_output(stdout) → TestResult
    - cleanup_container(container_id) → None

  ---

  PR AGENT SYSTEM PROMPT:
  --------------------------------
  You are a GitHub automation agent. Your task is to create a well-
  structured Pull Request for a code fix that has already passed all tests.

  Generate a PR description in this format:
    ## Summary
    [One paragraph describing what was fixed and why]

    ## Changes Made
    - [File]: [What changed]

    ## Test Evidence
    - Tests run: {total_tests}
    - Tests passed: {passed}
    - Test command: `pytest tests/`

    ## Related Issue
    Closes #{issue_number}

  PR AGENT TOOLS:
    - create_branch(owner, repo, branch_name, base_branch="main") → str
    - commit_files(owner, repo, branch, file_changes, message) → str
    - open_pull_request(owner, repo, title, body, head, base) → dict
      Returns: {pr_url, pr_number}

------------------------------------------------------------------------
2.5 REAL-TIME COMMUNICATION PROTOCOL
------------------------------------------------------------------------

  ARCHITECTURE: LangGraph → Event Queue → FastAPI WS → React Client

  STEP 1: Event Emission from LangGraph
  ----------------------------------------
  Each LangGraph node function receives a shared asyncio.Queue (injected
  via the config["configurable"] mechanism). Nodes call:

    await config["configurable"]["event_queue"].put({
        "type":      "agent_log",
        "run_id":    state["run_id"],
        "agent":     "research",
        "level":     "info",
        "message":   "Reading file: src/models/user.py",
        "timestamp": datetime.utcnow().isoformat()
    })

  EVENT TYPES:
    - "agent_log"       → A log line from an agent (most common)
    - "node_start"      → An agent node has begun execution
    - "node_complete"   → An agent node has finished
    - "state_update"    → Full state snapshot (sent after each node)
    - "diff_ready"      → Coding agent produced a patch (includes diff)
    - "test_result"     → Testing agent results summary
    - "pr_created"      → PR URL available
    - "run_failed"      → Terminal failure event
    - "run_complete"    → Successful completion event

  STEP 2: FastAPI WebSocket Handler
  ------------------------------------
  # File: backend/api/websocket.py

  @router.websocket("/ws/{run_id}")
  async def websocket_endpoint(websocket: WebSocket, run_id: str):
      await websocket.accept()
      
      # Retrieve the event queue for this run
      event_queue = run_manager.get_queue(run_id)
      if not event_queue:
          await websocket.close(code=4004, reason="Run not found")
          return
      
      try:
          while True:
              # Wait for event from LangGraph (with timeout for ping)
              try:
                  event = await asyncio.wait_for(
                      event_queue.get(),
                      timeout=30.0
                  )
              except asyncio.TimeoutError:
                  # Send keepalive ping
                  await websocket.send_json({"type": "ping"})
                  continue
              
              # Send event to client
              await websocket.send_json(event)
              
              # Terminal events: close connection cleanly
              if event["type"] in ("run_complete", "run_failed"):
                  break
                  
      except WebSocketDisconnect:
          # Client disconnected — the LangGraph task continues running
          # in the background. State is preserved in checkpointer.
          run_manager.mark_disconnected(run_id)
      finally:
          await websocket.close()

  STEP 3: React Client WebSocket Hook
  ---------------------------------------
  // File: frontend/src/hooks/useAgentStream.ts

  export function useAgentStream(runId: string | null) {
    const [logs, setLogs] = useState<AgentLog[]>([]);
    const [state, setState] = useState<AgentState | null>(null);
    const [status, setStatus] = useState<"idle"|"running"|"done"|"failed">();
    const wsRef = useRef<WebSocket | null>(null);

    useEffect(() => {
      if (!runId) return;

      const connect = () => {
        const ws = new WebSocket(`ws://localhost:8000/ws/${runId}`);
        wsRef.current = ws;

        ws.onmessage = (event) => {
          const data = JSON.parse(event.data);
          switch (data.type) {
            case "agent_log":
              setLogs(prev => [...prev, data]);
              break;
            case "state_update":
              setState(data.state);
              break;
            case "run_complete":
              setStatus("done");
              break;
            case "run_failed":
              setStatus("failed");
              break;
          }
        };

        ws.onclose = (event) => {
          if (event.code !== 1000 && status === "running") {
            // Reconnect after 2 seconds on unexpected close
            setTimeout(connect, 2000);
          }
        };
      };

      connect();
      return () => wsRef.current?.close(1000);
    }, [runId]);

    return { logs, state, status };
  }

2.7 ERROR HANDLING & LIMITS
------------------------------------------------------------------------

  INFINITE LOOP PREVENTION:
    MAX_RETRIES = 3  (configurable via env var AGENT_MAX_RETRIES)

    After MAX_RETRIES coding attempts, the graph routes to END with:
      state["final_status"] = "failed"
      state["error_message"] = "Max retries exceeded. Tests did not pass
                                 after {MAX_RETRIES} attempts."
    A "run_failed" WebSocket event is emitted with the last test output.

  PER-NODE TIMEOUTS:
    - Research Agent:  120 seconds (large repos may require many API calls)
    - Coding Agent:    60 seconds per LLM call
    - Testing Agent:   180 seconds (Docker build + test run)
    - PR Agent:        30 seconds

  WEBSOCKET DISCONNECTION HANDLING:
    - If client disconnects mid-run, the LangGraph graph CONTINUES running
      in the background (async task is independent of WS connection).
    - The event queue buffers up to 1000 events per run.
    - On client reconnect (same run_id), buffered events are replayed
      in order before switching to live streaming.
    - If the WS handler crashes, it does not affect the graph execution.

  LLM API ERROR HANDLING:
    - Exponential backoff with 3 retries on rate limit (429) errors.
    - On non-retriable errors (400, 401), node returns error state
      immediately and graph routes to END.

  DOCKER FAILURE HANDLING:
    - If Docker daemon is unreachable: fail fast with clear error message.
    - If container OOM-killed: TestResult.status = "error",
      stderr = "Container killed: memory limit exceeded".
    - Container is always removed (docker rm --force) in a finally block.

  def route_after_testing(state: AgentState) -> str
    """Conditional edge router after the Testing node."""
    test_result = state.get("test_result")
    if test_result is None:
        return "fail"
    if test_result["status"] == "pass":

FILE: backend/graph/orchestrator.py
--------------------------------------
Assemble the complete LangGraph graph:

  from langgraph.graph import StateGraph, END
  from langgraph.checkpoint.memory import MemorySaver

  Import all node functions:
    from agents.research_agent import research_node
    from agents.coding_agent import coding_node
    from agents.testing_agent import testing_node
    from agents.pr_agent import pr_node
    from graph.router import route_after_testing, route_after_research
    from core.state import AgentState

  def build_graph() -> CompiledGraph:
    graph = StateGraph(AgentState)

    # Add all nodes
    graph.add_node("research", research_node)
    graph.add_node("coding",   coding_node)
    graph.add_node("testing",  testing_node)
    graph.add_node("pr",       pr_node)

    # Entry point
    graph.set_entry_point("research")

    # Add conditional edge after research (handles fatal research errors)
    graph.add_conditional_edges(
        "research",
        route_after_research,
        {"continue": "coding", "fail": END}
    )

    # Fixed edges
    graph.add_edge("coding", "testing")

    # Conditional edges after testing (the self-correction loop)
    graph.add_conditional_edges(
        "testing",
        route_after_testing,
        {
            "pass":  "pr",
            "retry": "coding",
            "fail":  END
        }
    )

    # Terminal edge
    graph.add_edge("pr", END)

    # Compile with in-memory checkpointer
    checkpointer = MemorySaver()
    return graph.compile(checkpointer=checkpointer)

  # Singleton instance
  compiled_graph = build_graph()

  async def run_graph(initial_state: AgentState,
                       event_queue: asyncio.Queue) -> AgentState:
    """Execute the compiled graph for a single run."""
    config = {
        "configurable": {
            "thread_id":    initial_state["run_id"],
            "event_queue":  event_queue,
            "workspace_dir": tempfile.mkdtemp(prefix=f"mao_{initial_state['run_id']}_")
        }
    }
    final_state = None
    async for state_snapshot in compiled_graph.astream(
        initial_state,
        config=config,
        stream_mode="values"
    ):
        final_state = state_snapshot
        # Emit full state update to WebSocket after each node
        await event_queue.put({
            "type":  "state_update",
            "state": dict(final_state),
            "run_id": initial_state["run_id"]
        })
    
    # Emit terminal event
    if final_state and final_state.get("final_status") == "success":
        await event_queue.put({"type": "run_complete",
                                "run_id": initial_state["run_id"],
                                "pr_url": final_state.get("pr_url")})
    else:
        await event_queue.put({"type": "run_failed",
                                "run_id": initial_state["run_id"],
                                "error": final_state.get("error_message")})
    
    return final_state

FILE: backend/api/websocket.py

Implement WebSocket endpoint using APIRouter:

  ws_router = APIRouter()

  @ws_router.websocket("/ws/{run_id}")
  async def websocket_endpoint(websocket: WebSocket, run_id: str):
    await websocket.accept()
    event_queue = run_manager.get_queue(run_id)
    
    if not event_queue:
        await websocket.close(code=4004, reason="Run not found")
        return
    
    run_manager.mark_connected(run_id)
    
    try:
        # Replay any buffered events first
        buffered = []
        while not event_queue.empty():
            try:
                buffered.append(event_queue.get_nowait())
            except asyncio.QueueEmpty:
                break
        for event in buffered:
            await websocket.send_json(event)
        
        # Live streaming loop
        while True:
            try:
                event = await asyncio.wait_for(
                    event_queue.get(),
                    timeout=30.0
                )
                await websocket.send_json(event)
                if event.get("type") in ("run_complete", "run_failed"):
                    break
            except asyncio.TimeoutError:
                await websocket.send_json({"type": "ping",
                                            "run_id": run_id})
                
    except WebSocketDisconnect:
        run_manager.mark_disconnected(run_id)
    except Exception as e:
        await websocket.send_json({"type": "error", "message": str(e)})
    finally:
        try:
            await websocket.close()
        except Exception:
            pass
